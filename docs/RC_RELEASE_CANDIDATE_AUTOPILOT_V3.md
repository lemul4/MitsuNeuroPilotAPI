## 1. Выпуск релиз-кандидата программного продукта

### 1.1 Подготовка RC-версии с полной заявленной функциональностью

RC-версия фиксирует и предоставляет следующий функциональный контур:

1. Подготовка среды выполнения (conda-lock, requirements, editable install).
2. Закрытопетлевая оценка (closed-loop) нейросетевого агента в CARLA через обертку leaderboard.
3. Поддержка инференс-контура SensorAgent для выполнения маршрутов и формирования управляющих команд.
4. Сохранение артефактов прогона:
   - видео (debug/demo/input по конфигурации),
   - метрики и инфракции,
   - endpoint/json-отчеты.
5. Инструменты анализа результатов (web-интерфейс инфракций).
6. i-MiEV GUI для операторского контроля:
   - визуализация телеметрии и управляющих воздействий,
   - режимы ручного/виртуального/AI-управления,
   - интеграция с LEAD-агентом для демонстрации контуров управления.

Архитектурная специфика дипломного агента в RC:

1. Основа: подход класса TransFuser v6 (многокамерное end-to-end управление).
2. Сенсорный состав агента: только 3 RGB-камеры.
3. LiDAR и Radar в режиме дипломного агента не используются как входы политики.
4. Оптическая схема камер:
   - центральная фронтальная камера с более узким FOV,
   - две боковые камеры с более широким FOV, развернутые влево/вправо,
   - зоны обзора боковых камер пересекаются с зоной фронтальной камеры, что создает условия для извлечения квазистерео-признаков глубины на уровне эмбеддингов.

### 1.2 Обеспечение стабильности работы

Для RC фиксируется следующий набор мер стабильности:

1. Конфигурационная воспроизводимость:
   - единая спецификация зависимостей (requirements.txt, conda-lock.yml),
   - воспроизводимый запуск через скрипты scripts/.
2. Изоляция окружения:
   - запуск в выделенном окружении conda (env: lead),
   - фиксация переменных LEAD_PROJECT_ROOT и SCENARIO_RUNNER_ROOT.
3. Контроль совместимости:
   - Python >= 3.10,
   - CARLA 0.9.15,
   - Linux-платформа.
4. Наблюдаемость выполнения:
   - логирование в outputs/,
   - сохранение метрик, инфракций и видеоартефактов,
   - возможность пост-анализа в infraction dashboard.

Критерии приемки стабильности RC:

1. Скрипты установки выполняются без критических ошибок.
2. Один демонстрационный closed-loop маршрут проходит до конца без падения процесса агента.
3. Формируются артефакты в outputs/local_evaluation/.
4. Метрики/инфракции сериализуются в JSON и доступны для анализа.

Фиксация релизной идентичности выполняется следующим образом:

### 1.4 Обеспечение запуска и демонстрации

Ниже приведен минимальный регламент демонстрации RC:

```bash
# 1) Подготовка окружения
pip install conda-lock
conda-lock install -n lead conda-lock.yml
conda activate lead
pip install uv
uv pip install -r requirements.txt
uv pip install -e .

# 2) Подготовка CARLA
bash scripts/setup_carla.sh

# 3) Запуск симулятора
bash scripts/start_carla.sh

# 4) Прогон агента на маршруте (пример)
python lead/leaderboard_wrapper.py \
  --checkpoint outputs/checkpoints/tfv6_resnet34 \
  --routes data/benchmark_routes/bench2drive/23687.xml \
  --bench2drive

# 5) Просмотр аналитики инфракций
python lead/infraction_webapp/app.py

# 6) Запуск i-MiEV GUI
python "i-MiEV GUI/main.py"
```

Ожидаемый результат демонстрации:

1. Запуск CARLA и инициализация leaderboard-пайплайна.
2. Выполнение маршрута агентом с выдачей управляющих воздействий (steer/throttle/brake).
3. Сохранение артефактов в outputs/local_evaluation/<route_id>/.
4. Визуальный анализ ошибок через веб-интерфейс.
5. Онлайн-мониторинг параметров управления и телеметрии в i-MiEV GUI.

## 3. Подготовка документации к релиз-кандидату

### 3.1 Сведения о назначении программы

Область применения:

1. Исследования и разработка алгоритмов автономного вождения в виртуальной среде CARLA.
2. Учебно-научные задачи по обучению, валидации и сравнению end-to-end нейросетевых агентов.
3. Подготовка воспроизводимых экспериментальных прогонов для дипломной работы.
4. Демонстрация и отладка контуров управления через i-MiEV GUI в режиме операторского наблюдения.

Классы решаемых задач:

1. Задачи поведенческого планирования и управления ТС на основе многокамерного визуального ввода.
2. Задачи следования маршруту в динамической городской сцене (учет участников движения, знаков, светофоров, геометрии дороги).
3. Задачи оценки качества в закрытом цикле (closed-loop) по маршрутным benchmark-сценариям.
4. Задачи разбора нарушений и деградаций (infraction analysis).
5. Задачи визуализации телеметрии, ручного тестирования управляющих команд и проверки реакции системы через GUI.

Используемые методы и подходы:

1. End-to-end управление на основе глубокой нейронной сети.
2. Архитектурные идеи SOTA-линейки TransFuser v6 (мультимодальные/многокамерные представления и прогноз управляющих воздействий).
3. Многокамерная экстракция признаков:
   - центральная узкоугольная камера усиливает дальнодействующее восприятие по направлению движения,
   - боковые широкоугольные камеры расширяют периферийный обзор,
   - перекрытие полей зрения позволяет модели извлекать признаки глубины из межкамерной согласованности.
4. Постобработка управления и эвристики устойчивости в closed-loop контуре.

Ограничения и условия применения:

1. Система предназначена для симулятора CARLA и не является готовой системой для реального дорожного применения.
2. Достоверность результатов ограничена качеством симуляции, сценариев и доменным сдвигом sim-to-real.
3. Необходимо наличие совместимого GPU для практической скорости инференса/обучения.
4. Корректная работа зависит от согласованных версий CARLA, Python и зависимостей проекта.
5. При выпуске RC допускаются известные ограничения производительности и покрытия тестами, не влияющие на демонстрацию полного функционального контура.

### 3.2 Сведения о технических характеристиках

Минимальная конфигурация технических средств:

1. ОС: Linux x86_64.
2. CPU: 8+ логических потоков.
3. RAM: от 32 ГБ.
4. GPU: NVIDIA, от 12 ГБ VRAM (рекомендуется 16+ ГБ).
5. Диск: от 100 ГБ свободного пространства (с учетом CARLA, кэшей и артефактов).
6. Сеть: доступ к репозиториям зависимостей и checkpoint-хранилищу.

Рекомендуемая конфигурация для устойчивой демонстрации:

1. CPU: 12+ потоков.
2. RAM: 64 ГБ.
3. GPU: 24 ГБ VRAM.
4. SSD NVMe для ускорения чтения датасетов и записи артефактов.

Распределение функций между программной и аппаратной реализацией:

1. Программная часть:
   - загрузка и предобработка данных сенсоров,
   - инференс нейросети,
   - расчет управляющих команд,
   - логирование и пост-анализ,
   - операторский интерфейс i-MiEV GUI (Qt/PySide6, асинхронный обмен телеметрией).
2. Аппаратная часть:
   - GPU: ускорение вычислений нейросети,
   - CPU: orchestration, обработка пайплайна, сериализация логов,
   - RAM/диск: буферизация, кэширование, хранение артефактов,
   - опциональный последовательный интерфейс для связи с внешним контуром управления i-MiEV.

Используемые аппаратные и программные интерфейсы:

1. Программные интерфейсы:
   - CARLA Python API,
   - Python CLI (скрипты запуска/оценки),
   - файловый интерфейс ОС (чтение конфигов/датасетов, запись результатов),
   - локальный HTTP-интерфейс для infraction webapp,
   - GUI-интерфейс на базе PySide6 (i-MiEV GUI),
   - асинхронный последовательный обмен (serial/serial_asyncio) для телеметрии и команд.
2. Аппаратные интерфейсы:
   - CUDA-совместимое GPU (через PyTorch),
   - стандартные интерфейсы хранения данных (SSD/HDD).

Описание входных и выходных данных:

1. Входные данные:
   - последовательности RGB-кадров с 3 камер,
   - служебная телеметрия (скорость, инерциальные/навигационные данные симулятора),
   - маршрутные файлы и конфигурации benchmark-сценариев,
   - параметры модели и чекпоинты,
   - входные команды и параметры оператора из i-MiEV GUI (при соответствующем режиме).
2. Выходные данные:
   - управляющие команды агента (steer, throttle, brake),
   - маршрутные и агрегированные метрики,
   - журналы инфракций и диагностические JSON,
   - видео и изображения для визуальной верификации,
   - итоговые артефакты в outputs/local_evaluation и смежных каталогах outputs/,
   - поток оперативной телеметрии и статусы управления, отображаемые в i-MiEV GUI.

### 3.3 Требования по обеспечению безопасности

Требования к защите данных:

1. Ограничение доступа к рабочему окружению и каталогу проекта по принципу минимальных привилегий.
2. Использование изолированного Python-окружения (conda) для снижения риска конфликтов и подмены зависимостей.
3. Контроль источников моделей/датасетов (доверенные репозитории, контроль целостности загруженных файлов).

Требования к функциональной безопасности контура управления SensorAgent:

1. Безопасная инициализация управления:
   - в начальные кадры после старта агент принудительно держит полное торможение,
   - это исключает неконтролируемое движение в период инициализации состояния.
2. Защита от неконсистентного продольного управления:
   - при активном торможении дроссель сбрасывается в ноль,
   - при почти нулевой скорости также обнуляется угол для предотвращения накопления ошибок интегрального звена.
3. Защита от аномалий маршрутных целевых точек:
   - применяется адаптивная логика pop-distance в плотных участках,
   - слишком удаленная следующая target-point может быть отброшена как недостоверная.
4. Эвристика безопасного преодоления застревания:
   - режим force-move включается только после порогового времени стояния,
   - перед подачей «вытягивающего» газа выполняется проверка safety-box в ближней зоне по LiDAR-буферу,
   - при обнаружении препятствия force-move блокируется, включается торможение.
5. Контроль знака STOP в постобработке:
   - при конфигурационном включении активируется снижение скорости/остановка у STOP,
   - используются cooldown-счетчики, чтобы избежать колебаний решения на соседних кадрах.
6. Наблюдаемость safety-событий:
   - SensorAgent журналирует infractions по шагам и сохраняет их в JSON,
   - метрики и диагностические признаки (stuck_detector, force_move, brake и др.) сериализуются для пост-анализа.

Требования безопасности для i-MiEV GUI:

1. Защита от несанкционированного включения автоконтроля:
   - AI-контур запускается только при одновременном выполнении условий: включен чекбокс AI, активирован Control Active, есть подключение (виртуальное или serial).
2. Ограничение управляющих воздействий в интерфейсе:
   - газ, тормоз и угол руля ограничиваются программно (клиппинг диапазонов),
   - в обработке клавиш реализовано правило «тормоз отменяет газ».
3. Целостность CAN-команд:
   - перед отправкой команд пересчитывается CRC8,
   - команды отправляются через безопасную очередь, что снижает риск гонок и неконсистентных пакетов.
4. Безопасная деградация при потере связи:
   - при disconnect/connection_lost интерфейс переводится в состояние отключения,
   - отключается управление и останавливается связанный контур.
5. Режим безопасной демонстрации:
   - предусмотрен VIRTUAL_DEMO_MODE для отладки без физического воздействия на внешний исполнительный контур.

Меры по обеспечению конфиденциальности и целостности информации:

1. Разделение данных экспериментов и служебных логов по каталогам.
2. Регулярное резервное копирование критичных артефактов (конфиги, чекпоинты, отчеты).
3. Фиксация версий зависимостей и конфигурации запуска для воспроизводимости и целостности экспериментов.
4. Ограничение публикации внутренних логов, если они содержат чувствительную инфраструктурную информацию.
5. Для telemetry JSONL/CSV применять контроль прав доступа и ротацию файлов, так как логи содержат детальные управляющие сигналы (steer/throttle/brake).

Механизмы аутентификации и авторизации (при наличии):

1. В локальном стенде проекта специализированная серверная подсистема authN/authZ не требуется.
2. Аутентификация и авторизация реализуются средствами ОС, контроля доступа к репозиторию и инфраструктуре CI/CD (при использовании).

Ограничения доступа и меры защиты от несанкционированного использования:

1. Ограничить права записи в каталоги модели и результатов для непроверенных пользователей.
2. Не хранить секреты (токены/ключи) в открытом виде в репозитории.
3. Применять проверку изменений перед релизом (code review, линтеры, статический анализ).
4. Использовать отдельные учетные записи/окружения для разработки и демонстрации RC.
5. Для i-MiEV GUI ограничивать доступ к serial-порту доверенными пользователями и проверять выбранный порт перед отправкой управляющих команд.
6. В демонстрационном стенде физического управления включать последовательность «Connect -> Control Active -> AI», запрещая обход этого порядка.

Подтвержденные в коде механизмы безопасности (SensorAgent и i-MiEV GUI):

1. SensorAgent:
   - Safe-start с принудительным тормозом на старте,
   - постобработка ForceMovePostProcessor с safety-box emergency stop,
   - постобработка StopSignPostProcessor,
   - логирование infractions и сохранение safety-диагностики.
2. i-MiEV GUI:
   - gating запуска AI по состоянию подключения и активации управления,
   - ограничение диапазонов управляющих команд,
   - приоритизация торможения над акселерацией,
   - CRC8 и очередность передачи serial-команд,
   - виртуальный безопасный режим тестирования.

## Приложение B. Формулировка результата дипломной работы

Результатом дипломной работы является программная система обучения и оценки нейросетевого агента автономного управления транспортным средством в симуляторе CARLA, реализующая end-to-end подход на базе идей TransFuser v6 в конфигурации визуального восприятия с 3 камерами (без LiDAR и Radar), с поддержкой воспроизводимого запуска, тестирования и анализа качества в closed-loop сценариях, а также с операторским контуром мониторинга и управления через i-MiEV GUI.
