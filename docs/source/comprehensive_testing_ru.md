- Наименование проекта: программная система обучения нейросетевых агентов автономного управления транспортными средствами на базе симулятора CARLA.
- Репозиторий: текущий проект LEAD с адаптацией архитектуры TransFuser v6.

## 1. Изучение и анализ предмета тестирования

### 1.1 Объект тестирования

Объектом тестирования является программная система, реализующая полный цикл:

1. Подготовку и загрузку данных для обучения агента.
2. Обучение и инференс нейросетевого агента автономного вождения.
3. Закрытую оценку в симуляторе CARLA через Leaderboard-совместимый интерфейс.
4. Сбор диагностических артефактов (метрики, журналы, видео, инфракции).

Ключевая особенность рассматриваемой конфигурации агента:

- Архитектурная база: TransFuser v6 (адаптированная версия).
- Сенсорный стек для основного эксперимента диплома: только 3 RGB-камеры.
- Геометрия камер: одна фронтальная камера с относительно узким FOV, две боковые с более широким FOV и разворотом в стороны, с частичным перекрытием углов обзора с фронтальной камерой.
- Цель такого стека: обеспечить сети возможность извлекать признаки глубины и пространственного согласования за счет перекрывающихся зон наблюдения (квази-стерео эффект без LiDAR/RADAR).

### 1.2 Основные функциональные возможности продукта

1. Запуск агента в закрытом цикле CARLA (модуль `lead/inference/sensor_agent.py`).
2. Унифицированный запуск оценок в разных режимах leaderboard (модуль `lead/leaderboard_wrapper.py`).
3. Конфигурирование логики закрытого цикла (модуль `lead/inference/config_closed_loop.py`).
4. Формирование сенсорной схемы и калибровок (модуль `lead/common/sensor_setup.py`, калибровки в `lead/common/config_base.py`).
5. Планирование траектории/waypoints/target-speed декодером (модуль `lead/tfv6/planning_decoder.py`).
6. Формирование артефактов тестирования: `checkpoint_endpoint.json`, `debug_checkpoint`, журналы выполнения, инфракции, диагностические видео.

### 1.3 Компоненты и сценарии, подлежащие тестированию

#### Компоненты

1. Запуск и окружение:
- корректная активация conda-окружения;
- корректная установка переменных среды (`LEAD_PROJECT_ROOT`, `SCENARIO_RUNNER_ROOT`, `PYTHONPATH`).

2. Маршрутизация оценочного запуска:
- корректный выбор режима STANDARD/BENCH2DRIVE/AUTOPILOT;
- корректный выбор evaluator-скрипта и agent-файла;
- корректный вывод директорий результатов.

3. Сенсорная подсистема (фокус диплома):
- корректная регистрация 3 камер;
- корректные параметры FOV/rotation;
- корректная обработка изображений и отбор используемых камер.

4. Нейросетевой инференс:
- корректная загрузка checkpoint/config;
- корректная генерация route/waypoints/target speed;
- устойчивость к вариативности сценариев.

5. Постобработка и безопасность:
- работа эвристик (стоп-знак, stuck/creeping, корректоры);
- отслеживание и логирование нарушений (infractions).

6. Результаты и отчетность:
- корректная генерация checkpoint отчета;
- корректная фиксация статуса маршрутов, метрик и диагностической информации.

#### Сценарии

1. Базовый smoke-запуск одного маршрута.
2. Запуск в режиме Bench2Drive.
3. Запуск в режиме expert/autopilot для эталонной проверки инфраструктуры.
4. Проверка корректности сенсорного профиля «3 камеры».
5. Проверка устойчивости к ошибкам конфигурации окружения.

### 1.4 Как должны работать тестируемые функции

1. При валидной конфигурации и доступном CARLA-сервере система должна завершать маршрут без аварийного падения процесса.
2. Wrapper должен корректно формировать команды и окружение для нужного leaderboard-режима.
3. Агент должен принимать RGB-поток, выполнять инференс и выдавать управляющие команды в каждом тике симуляции.
4. По завершении должен формироваться машиночитаемый итог (`checkpoint_endpoint.json`) с полями статуса, прогресса, score-метрик, инфракций.
5. При ошибках конфигурации должны формироваться диагностические сообщения в логах.

### 1.5 Open-loop, Closed-loop и benchmark-контур автопилота

В проекте целесообразно разделять три класса проверки поведения системы управления.

1. Open-loop (оффлайн оценка предсказаний без замыкания контура управления):
- Назначение: проверить качество выходов модели (траектория, waypoints, target speed) на заранее собранных данных без влияния ошибок управления на будущие наблюдения.
- Конфигурационный уровень: `lead/inference/config_open_loop.py`.
- Ключевые параметры open-loop: снижение целевой скорости (`lower_target_speed`, `lower_target_speed_factor`), порог торможения (`brake_threshold`), строгая загрузка весов (`strict_weight_load`).
- Практический смысл для ВКР: быстрое сравнение гипотез по архитектуре 3-камерного encoder-decoder до дорогих closed-loop прогонов.

2. Closed-loop (онлайн оценка в CARLA):
- Назначение: проверить фактическое поведение агента в контуре perception-planning-control при обратной связи среды.
- Конфигурационный уровень: `lead/inference/config_closed_loop.py`.
- Ключевые параметры closed-loop: контроль модальностей управления (`steer_modality`, `throttle_modality`, `brake_modality`), эвристики stuck/stop-sign, параметры PID-контроллеров, генерация debug/demo артефактов.
- Практический смысл для ВКР: именно closed-loop определяет реальную состоятельность 3-камерной схемы и устойчивость на маршрутах.

3. Benchmark автопилота (expert/autopilot baseline):
- Назначение: получить эталонный baseline для инфраструктуры запуска, корректности сценариев и верхней оценки качества маршрута.
- Технически реализуется через autopilot evaluator и expert-агента.
- В проекте используется как контрольный прогон, с которым сравнивается нейросетевой агент.

### 1.6 Подсистема i-MiEV GUI

`i-MiEV GUI` в данном репозитории является отдельным эксплуатационным модулем для взаимодействия с контуром управления и телеметрией.

Функциональная роль:

1. GUI-панель оператора для мониторинга состояния управления (скорость, угол руля, передачи, педали).
2. Асинхронная работа с последовательным портом и CAN-пакетами.
3. Интеграция с LEAD-автопилотом через отдельный поток запуска (`LeadAgentThread`) и прокидывание переменных окружения.
4. Чтение/запись telemetry-потока и визуализация сигналов.

Основные компоненты i-MiEV GUI:

1. Главное окно и логика UI: `i-MiEV GUI/main.py`.
2. Интеграция запуска LEAD: `i-MiEV GUI/lead_integration.py`.
3. Константы канала и частоты: `i-MiEV GUI/config.py`.
4. Парсинг входящих CAN-данных: `i-MiEV GUI/parser.py`.
5. Пакетирование, CRC8 и буфер: `i-MiEV GUI/utils.py`.

Критичные требования к тестированию GUI:

1. Корректность декодирования входных пакетов и защит от битых/неполных кадров.
2. Отсутствие блокировок UI при активной serial-коммуникации.
3. Корректное завершение/остановка дочернего процесса автопилота.
4. Консистентность телеметрии GUI и телеметрии, выдаваемой LEAD-агентом.

## 2. Планирование тестирования

### 2.1 Цели и объем тестирования

Цели:

1. Подтвердить работоспособность программного продукта как целостной системы.
2. Подтвердить корректность выполнения ключевых пользовательских сценариев дипломной работы.
3. Оценить устойчивость к типовым эксплуатационным ошибкам (окружение, пути, переменные среды).
4. Зафиксировать качество и полноту тестовой документации для защиты ВКР.

Объем:

- Включено: функциональные, интеграционные, приемочные, эксплуатационные сценарии на уровне запуска, инференса и формирования отчетов.
- Не включено: длительные массовые прогоны на сотнях маршрутов и стресс-тесты многосуточной стабильности (выделены как пост-дипломные задачи).

### 2.2 Выбранные виды тестирования

1. Функциональное тестирование.
2. Интеграционное тестирование.
3. Приемочное тестирование.
4. Эксплуатационное тестирование.

### 2.3 Тестовая стратегия

Стратегия комбинированная:

1. Risk-based подход:
- Приоритет 1: сценарии запуска и оценки в CARLA, так как они критичны для конечной демонстрации.
- Приоритет 2: корректность сенсорной конфигурации 3 камер.
- Приоритет 3: полнота логирования и отчетных артефактов.

2. Incremental end-to-end:
- Сначала проверяется окружение и конфигурация.
- Затем одиночный маршрут как smoke.
- Затем расширенные сценарии с проверкой метрик и журналов.

3. Requirements traceability:
- Каждый тест связан с конкретным функциональным требованием.

### 2.4 Критерии начала и завершения тестирования

Критерии начала:

1. Репозиторий в согласованной версии.
2. Активировано окружение `lead`.
3. Доступен CARLA и корректно задана конфигурация путей.
4. Подготовлены checkpoint и routes для тестовых прогонов.

Критерии завершения:

1. Выполнены все тесты из обязательного набора (critical/high).
2. Нет открытых дефектов критичности Critical и High, блокирующих демонстрационный сценарий.
3. Сформирован пакет документов: отчет, журнал испытаний, отчет о дефектах.

### 2.5 Программные и аппаратные средства

Программные:

1. ОС: Linux.
2. Python: 3.10+.
3. Conda-окружение: `lead`.
4. CARLA 0.9.15.
5. Подсистемы: leaderboard/scenario_runner (включая autopilot и Bench2Drive-ветки).
6. ffmpeg (для видео-артефактов).

Аппаратные:

1. CPU x86_64, не менее 8 потоков (рекомендовано).
2. GPU NVIDIA с поддержкой CUDA (рекомендовано для инференса).
3. RAM не менее 32 GB (рекомендовано для стабильной симуляции + инференса).
4. SSD для checkpoint, логов и видео.

## 3. Подготовка тестовой документации

## 3.1 Введение (принципы и подходы)

Примененные принципы тестирования:

1. Полнота по жизненному циклу: от запуска до оценки результатов.
2. Репрезентативность: сценарии выбраны из реальных режимов использования дипломной системы.
3. Воспроизводимость: каждый тест имеет четкие входные данные, шаги, ожидаемые результаты и артефакты.
4. Прослеживаемость: тесты связаны с целями и функциональными блоками.
5. Практическая применимость: процедуры подходят для повторного прогона перед защитой и перед релизом.

### 3.2.1 Объект тестирования

Единый программный комплекс обучения/инференса/оценки нейросетевого агента автономного вождения в CARLA.

- проверка устойчивости к типовым конфигурационным ошибкам.

- I-03: корректное взаимодействие agent + scenario_runner + leaderboard.

#### Группа A (Acceptance)

- A-01: демонстрационный маршрут завершается без критического сбоя.
- A-02: итоговые метрики и статус маршрута доступны в отчете.
- A-03: сенсорная конфигурация соответствует целевой архитектуре ВКР (3 камеры).
- A-04: baseline автопилота в benchmark-контуре формирует валидный итоговый отчет.

#### Группа O (Operational)

- O-01: проверка поведения при отсутствующей переменной `SCENARIO_RUNNER_ROOT`.
- O-02: повторяемость запуска на одинаковом маршруте.
- O-03: проверка качества диагностических журналов при ошибочных параметрах.
- O-04: устойчивость i-MiEV GUI к обрывам serial-соединения/перезапуску процесса автопилота.

## 3.5 Тестовые процедуры

### TP-01: Smoke-тест single route

- Цель: подтвердить базовую работоспособность полного контура.
- Условия запуска:
  1. Активно окружение `lead`.
  2. Запущен CARLA сервер.
  3. Подготовлены checkpoint и route-файл.
- Входные данные:
  1. Путь к checkpoint директории.
  2. Путь к route XML.
- Шаги:
  1. Запустить CARLA (`scripts/start_carla.sh`).
  2. Запустить wrapper/leaderboard на одном маршруте.
  3. Дождаться завершения эпизода.
  4. Проверить наличие итогового JSON-отчета.
- Ожидаемые результаты:
  1. Процесс завершен без аварийного исключения.
  2. `entry_status` в отчете: `Finished`.
  3. Поля score и status присутствуют.
- Выходные данные и журналы:
  1. `checkpoint_endpoint.json`.
  2. Логи stdout/stderr запуска.
  3. Диагностические файлы debug_checkpoint (если включены).

### TP-02: Проверка целевого сенсорного профиля (3 камеры)

- Цель: подтвердить соответствие архитектуры агента требованиям ВКР.
- Условия запуска:
  1. Конфигурация `TargetDataset.CARLA_LEADERBOARD2_ONLY3CAMERAS` или эквивалентная.
- Входные данные:
  1. Значения `camera_calibration`.
- Шаги:

### 4.4 Выявленные дефекты и отклонения

Обнаруженные проблемы:

1. DEFECT-GUI-PLATFORM-001 (High)
- Область: `i-MiEV GUI/lead_integration.py`.
- Суть: жесткая ориентация путей на Windows (`venv/Lib/site-packages`, `venv/Scripts/python.exe`) в проекте, запускаемом в Linux.
- Последствие: риск некорректного запуска LEAD из GUI на Linux.
- Влияние на тесты: I-05, O-04, TP-07.

2. DEFECT-OPS-ROUTEHOST-001 (Medium)
- Область: `scripts/eval_expert.sh`.
- Суть: жестко заданный `--host=172.30.96.1`.
- Последствие: запуск падает или не подключается на стендах с другой сетевой топологией.
- Влияние на тесты: TP-06, O-01.

3. DEFECT-MODEL-FRUSTUM-001 (Medium)
- Область: `lead/common/common_utils.py`.
- Суть: при `num_used_cameras >= 3` используется упрощающее правило "всё во фрустуме".
- Последствие: потенциальная переоценка видимости объектов и искажение аналитики в edge-case сценариях.
- Влияние на тесты: TP-02, TP-04.

4. DEFECT-ENV-001 (Medium)
- Область: запуск route evaluation.
- Суть: при неполной конфигурации окружения (`SCENARIO_RUNNER_ROOT`, `PYTHONPATH`) возможны пропуски/ошибки сценариев.
- Последствие: нестабильная воспроизводимость на новом стенде.
- Влияние на тесты: O-01, I-03, TP-05.

## 5.3 Отчет о выявленных дефектах

### DEFECT-GUI-PLATFORM-001

- Тип: кроссплатформенный дефект интеграции.
- Описание: в GUI-интеграции используются Windows-пути и предполагаемая структура виртуального окружения, что неуниверсально для Linux.
- Условия возникновения: запуск `LeadAgentThread` в Linux окружении.
- Критичность: High.
- Статус устранения: открыт.
- Рекомендация: использовать кроссплатформенный резолвер путей (`sys.executable`, `site.getsitepackages`, проверка `os.name`).

### DEFECT-OPS-ROUTEHOST-001

- Тип: эксплуатационный дефект конфигурации.
- Описание: hard-coded host в `eval_expert.sh` ухудшает переносимость тестов между стендами.
- Условия возникновения: несоответствие IP текущему CARLA-хосту.
- Критичность: Medium.
- Статус устранения: открыт.
- Рекомендация: вынести host в переменную окружения с дефолтом `localhost`.

### DEFECT-MODEL-FRUSTUM-001

- Тип: методологический/модельный риск.
- Описание: допущение о полном покрытии фрустума для 3+ камер может искажать фильтрацию объектов.
- Условия возникновения: анализ edge-case геометрий в open-loop/label filtering.
- Критичность: Medium.
- Статус устранения: открыт.
- Рекомендация: заменить хак на реальную геометрию объединенного FOV по калибровкам.

### DEFECT-ENV-001

- Тип: конфигурационный дефект/эксплуатационный риск.
- Описание: неполная настройка `SCENARIO_RUNNER_ROOT` и `PYTHONPATH` приводит к нестабильному route evaluation.
- Условия возникновения: новый стенд без preflight-проверки окружения.
- Критичность: Medium.
- Статус устранения: частично смягчен fallback-логикой в wrapper.

| RQ-05 | Пригодность для демонстрации ВКР | A-01, A-02, раздел 6 |

## Приложение B. Рекомендуемый чек-лист перед очной демонстрацией

1. Активировано окружение `lead`.
2. Проверены `LEAD_PROJECT_ROOT` и `SCENARIO_RUNNER_ROOT`.
3. CARLA сервер запущен и доступен по host/port.
4. Выбран корректный checkpoint дипломного 3-camera агента.
5. Проверено наличие route-файла и права записи в output-каталог.
6. После прогона проверены `entry_status`, score-метрики, инфракции, диагностические артефакты.

## Приложение C. Контурная схема проверки open-loop/closed-loop/autopilot

1. Open-loop:
- Используется для быстрых итераций по качеству предсказаний модели на фиксированных данных.
- Не заменяет closed-loop, а служит фильтром гипотез перед дорогим запуском в CARLA.

2. Closed-loop:
- Основной критерий практической состоятельности агента в задаче автономного вождения.
- Обязателен для финальной оценки дипломного 3-камерного решения.

3. Autopilot benchmark:
- Нужен как инженерный baseline для сравнения с нейросетевым агентом.
- Позволяет отделить проблемы модели от проблем инфраструктуры запуска.

4. Минимальная последовательность прогонов перед защитой:
1. Open-loop smoke на небольшом наборе кадров.
2. Closed-loop single-route проверка.
3. Closed-loop benchmark (минимум один маршрут из целевого набора).
4. Autopilot baseline на том же/сопоставимом маршруте.
5. Сравнение score и инфракций, фиксация вывода в отчете.

## Приложение D. Эксплуатационное описание i-MiEV GUI

1. Назначение:
- Человеко-машинный интерфейс для мониторинга и управления контуром i-MiEV.
- Инструмент интеграции LEAD-автопилота с телеметрией и низкоуровневым каналом управления.

2. Технические характеристики (по текущей реализации):
- GUI стек: PySide6 + qasync + pyqtgraph.
- Serial: `serial_asyncio`, baudrate 1000000.
- Пакет: 16 байт, стартовый байт `0xAA`, контроль CRC8.
- Базовые сигналы парсера: скорость (`0x0003`), угол (`0x0001`), тормоз (`0x0017`), акселератор (`0x0018`), передача (`0x0004`).

3. Интеграция с LEAD:
- Запуск через `LeadAgentThread` с формированием окружения (`LEAD_PROJECT_ROOT`, `PYTHONPATH`, telemetry-переменные).
- Передача логов процесса в GUI и отображение статуса жизненного цикла автопилота.

4. Риски и контрольные меры:
- Риск рассинхронизации GUI и агентной телеметрии: устраняется периодической валидацией jsonl-потока.
- Риск зависания при разрыве порта: покрывается тестом O-04 и проверкой корректного reconnect.
- Риск платформенных отличий запуска процесса (Windows/Linux): покрывается отдельной проверкой stop-сценария.
